# 046《团结一切抗日力量，反对反共顽固派》对软件开发的启发：分类处理而非一刀切，用"十没有"立质量证据

## 原文出处
- 毛泽东《团结一切抗日力量，反对反共顽固派》，1940 年 2 月 1 日
- 原文链接：https://www.marxists.org/chinese/maozedong/marxist.org-chinese-mao-19400201.htm

## 原文核心观点（为软件工程师提炼）
1. 对不同类别对象应采取**不同政策**，不要扩大化/一刀切。
2. **两条政策并行**：打击破坏者 + 团结合作者，同时做。
3. 用**可验证的具体事实清单**（"十没有"）来立信用、反击污蔑。
4. 真整合必须建立在"**共同目标+互相尊重+双方成长**"三件事上，否则是假整合。

## 转译到软件开发的启发

### 启发一：代码审查/技术争论——别一棒子打翻

工程师常犯的"扩大化"错误：

- 看到一个 PR 写得烂 → "这个人整体技术不行"
- 看到一次架构决策有问题 → "这个组做啥都不靠谱"
- 看到一次线上故障 → "这套技术栈整体有毒"
- 看到一次代码评审冲突 → "这个 reviewer 就是挑刺的"
- 看到一个 bug → "这个库整体 bug 多"

每一次"扩大化判断"都会**让你错失合作机会**：
- 那个写烂 PR 的人，可能只是当时状态差，换个任务就很猛
- 那个架构决策有问题的组，可能在其他维度很能打
- 那套技术栈的故障，可能只是你们的使用姿势有问题

**毛的方法论翻译到工程**：
对**具体问题**坚决还击（review 打回/根因分析/故障复盘），
对**人/组/技术栈整体**保持默认合作+分类评估。

**口袋里的一句话**：
把"对事的严厉"和"对人的扩大化"分开——
对 PR 严厉、对代码严厉、对 bug 严厉，
但别把对事的严厉升级成"这个人/这个栈/这个组整体不行"。

### 启发二：两条政策并行——质量红线 + 积极扶持

毛："一方面反对……一方面团结……"

工程团队最常见的**单腿政策**：
- 只会抓问题不会扶持 → 团队恐惧文化，优秀的人跑光
- 只会鼓励不抓问题 → 质量底线崩塌，大家共同摆烂

正确做法是**两条政策并行**：

**"坚决还击"政策（对质量问题）**：
- PR 不过 standard 就是不过
- 测试不过就是卡发布
- 故障必须 RCA 文档
- 反复出问题的模块必须重构
- 长期不作为的 owner 必须换

**"积极团结"政策（对人和合作）**：
- 每周公开表扬 1-2 个好 PR
- 新人入职 30 天内必有 mentor 一对一
- 跨组合作主动送 context，不让对方自己啃
- 故障复盘对人不追责，对流程找问题
- 每季度有工程师骄傲 moment（技术分享/赛事/论坛）

**两条腿同时走**，团队才会既有底线又有温度。
只抓问题的团队会恐惧，只鼓励的团队会烂掉。

### 启发三：用"十没有"立信用——证据型工程文化

毛的"陕甘宁边区十没有"是**证据驱动传播**的教科书：
不说"我们很先进"，而是"一没有……二没有……"——
每一条都**可被现场查证**。

工程团队对外/对内传播常见的毛病：**只有形容词没有事实**。
典型的"空话型" README/对外宣传：
- "我们代码质量很高" ← 空话
- "我们流程很完善" ← 空话
- "我们团队很强" ← 空话

学毛的做法——**写"十没有/十有"清单**，每条可验证：

**好的工程文化 README 范例**：
1. 所有核心服务 >90% 测试覆盖，[链接到 Coveralls dashboard]
2. 过去 12 个月无 P0 故障，[链接到事件台账]
3. 所有 PR 平均合入 <6 小时，[链接到 PR metrics]
4. 所有生产变更必须经过 2 人 review，[链接到流水线配置]
5. 每个服务必有 oncall owner，[链接到 Pagerduty 表]
6. 每个新人 90 天内 ship 至少 3 个生产特性，[链接到 onboarding 看板]
7. 月度故障演练覆盖所有核心链路，[链接到演练报告]
8. 所有线上事件必有 48h RCA 文档，[链接到 RCA 库]
9. 核心服务 SLA 99.95%+，[链接到 SLO dashboard]
10. 技术债 ticket 每季消化 >=30%，[链接到 Jira burndown]

**每一条都是"可验证事实"，不是"自我评价"**。

对内复盘时也一样——不要说"这个季度我们进步很大"，
强制列出 5-10 条"过去没有、现在有"的具体事实。

### 启发四：真整合 vs 假整合——微服务合并、团队合并、架构统一

毛说："以消灭对方为代价的统一不是真统一。"

工程里的"假整合"非常常见：

- **微服务合并**："我们应该统一成一个大 monolith" → 实际上是某个 team 想把所有代码收编，对方被消灭
- **技术栈统一**："公司应该统一技术栈" → 实际上是 Java team 想搞掉所有非 Java 项目
- **数据库统一**："统一用 MySQL" → 实际上是否认其他场景的合理性
- **规范统一**："所有项目用一样的代码规范" → 实际上是一个 team 把自己的偏好强加给别人

毛的三问在工程里同样适用：
1. 是否有真正的共同目标？（"降低维护成本"是共同目标，"我们 team 扩权"不是）
2. 是否互相尊重各自独立性？（允许合理场景下保留差异，还是一定要全部消灭差异？）
3. 是否双方都在收益？（整合后双方都更好，还是一方吞并一方？）

**工程里的"假整合"有个典型特征**：
推动者只谈"统一后的一致性"，从不谈"消灭掉的那些方案当初为什么存在"。
真整合必须**尊重差异的来源**，然后判断哪些可以收敛、哪些必须保留。

## 给自己的三个自检问题
1. 我最近对某个同事/某个组/某个技术栈下的"整体判断"，是基于几件具体事实还是一次印象？
2. 我们团队现在"两条政策"齐全吗？既有质量底线（坚决还击），也有积极扶持（团结协作）？缺哪条？
3. 最近在我参与的"整合/统一"项目里，是真整合（共同目标+尊重差异+双方成长），还是某一方借"统一"之名扩权？
