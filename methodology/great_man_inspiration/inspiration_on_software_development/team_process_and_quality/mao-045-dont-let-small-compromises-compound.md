# 045《克服投降危险，力争时局好转》对软件开发的启发：别被"小问题"的孤立假象骗了

## 原文出处
- 毛泽东《克服投降危险，力争时局好转》，1940 年 1 月 28 日
- 原文链接：https://www.marxists.org/chinese/maozedong/marxist.org-chinese-mao-19400128.htm

## 原文核心观点（为软件工程师提炼）
1. 不要把各地发生的**投降、反共、倒退**现象孤立起来看，要识别其严重性。
2. 不要只从"**惧怕破裂统一战线**"一点设想而听任退让。
3. **两种方向同时存在**，两种前途都可能——积极行动者决定走向。
4. 具体行动纲领要细到 **11 条可执行口号**，而不仅仅是方针。

## 转译到软件开发的启发

### 启发一：技术债是"地方性事变"的集合——别孤立看待

毛说："不要把各地发生的投降、反共、倒退等严重现象孤立起来看。"
在软件开发里最贴切的对应就是**技术债和质量退化**。

工程师最常犯的错：

- "这次加个 if else 临时绕一下就行"
- "这个 test 先 skip，下次补"
- "这块代码先不重构，下个季度再说"
- "CI 偶尔 flaky，重试就好"
- "监控有 3 条告警经常响，大家都习惯了"

每一条单独看都"不是大事"，但**合起来是整个工程系统在向失控滑坡**：
- 临时 if 累积成意大利面 → 新人看不懂
- skip 的 test 累积成"我们已经不知道什么是对的"
- 拒绝重构累积成"这个模块没人敢动"
- flaky CI 累积成"所有人开始忽视 CI 信号"
- 习惯性告警累积成"真告警响起也没人看"

**识别方式**：
如果你在一个 sprint 里对 3 个以上"小技术妥协"点头，
停下来问——这些是不是同一条质量滑坡曲线上的几个点？

**口袋里的一句话**：单独的技术妥协不是大事，
密集出现的技术妥协是整个工程文化在崩塌的信号。

### 启发二：不要只从"怕 block 上线"一点做技术决策

毛说："如果只从惧怕破裂统一战线一点设想……"

软件团队里的翻版："如果只从惧怕 block 上线一点做决策……"

这种心态的典型表现：
- PR 有质量问题但 DDL 快到了 → 合
- 测试覆盖不够但"先上了再说" → 上
- 架构有明显问题但"这个 sprint 不能改了" → 拖
- 安全扫描报了高危但"这次先绕过，下版修" → 绕

每一次单独看都"为了按时交付"，累积起来就是**质量底线被系统性让步**。

毛的方案是"**双管齐下**"：
一方面坚决反抗（对核心质量问题不让步），
一方面积极扩大统一战线（说服产品/老板/利益相关者，让他们站到质量这一边）。

对应到工程实践：

- **对产品**：不是一味拒绝"上线压力"，而是主动把技术债/风险量化给他们看，让他们跟你一起决策
- **对老板**：不是只抱怨"没时间重构"，而是把重构的 ROI（减少故障数/提效 X%）算清楚
- **对同事**：不是背地里吐槽屎山，而是团结同样在意质量的人，一起推动标准

**口袋里的一句话**：
如果你的技术决策理由只剩"怕延期/怕被骂"，
多半是在把未来的稳定性卖给现在的安心。

### 启发三：两种前途同时存在——项目此刻在分叉

毛说："两种方向同时存在，两种前途都有可能。"

任何一个正在进行中的项目/系统/团队，此刻都处在这种"分叉状态"：

| 向好的前途 | 向坏的前途 |
|---|---|
| 架构越来越清晰，新人快速上手 | 代码越来越乱，新人不敢动 |
| 测试覆盖上升，发布越来越稳 | 测试衰减，每次发布都提心吊胆 |
| 监控看板越做越实用 | 告警疲劳，谁也不看了 |
| 团队文档沉淀越来越厚 | 知识全在几个老人脑子里 |
| 成员能互相 review 批评 | 团队越来越客气，问题没人点破 |

**关键判断**：你的项目现在在哪一列滑？
别骗自己"还行"，也别悲观"已经完了"——
**此刻的你和你的 team 做什么，决定倒向哪一边**。

具体动作：每个 sprint 结束花 30 分钟做一次"分叉体检"，
对以上若干维度各打一个方向箭头（↑/↓），
连续 3 个 sprint ↓ 的维度就是下个 sprint 必须干预的重点。

### 启发四：行动纲领要细到 11 条可执行口号

毛在这篇里最可操作的部分是末尾那 **11 条反汪口号**：
"(一)拥护抗战到底……(二)全国人民团结起来……(三)拥护国共合作……"

这不是高屋建瓴的战略，而是**可以直接印在传单上、喊在大会上、写进宣言里**的具体口号。

工程团队搞质量改进/架构演进时最大的问题：**停留在"方针"层面**，
比如"我们要提升代码质量"、"我们要重视技术债"、"我们要改善 CI"——
这些话一个季度后团队没人记得。

学毛这招——把方针炸开成**一组具体到可贴墙上的动作口号**：

例：团队"质量改进"不要写"我们要提升代码质量"，
应该写成：

1. PR 必须 >=1 人 review，超过 300 行拆分
2. 任何失败 test 禁止 skip，必须当场修或立票
3. 发布前必须跑一遍全量 smoke，红就卡发布
4. 每周一小时 pairing，强制跨人轮换
5. 故障必须 RCA 文档，不是口头复盘
6. 监控告警必须有 owner，无 owner 的就删
7. 所有硬编码字符串必须走常量文件
8. 新功能必须带 e2e test，不写的 PR 不合
9. 每周代码质量看板，一条一条读
10. 架构变更必须有 ADR 文档
11. 技术债 ticket 每个 sprint 必须消化 >=2 条

这 11 条每一条都能**直接挂在墙上，直接在 PR comment 里引用编号**。
这才叫**落地的纲领**。

**口袋里的一句话**：
任何"方针"到不了"可贴墙的编号清单"这一步，
都是漂在空中的口号。

## 给自己的三个自检问题
1. 我最近一个月合进去的 PR 里，有多少个"小妥协"？它们合起来是不是同一条质量滑坡？
2. 我上一次"为了不 block 上线"而放弃了某条质量红线，是什么时候？那次之后，这条红线还在吗？
3. 我们 team 的"质量改进方针"，能不能炸开成 10 条以上可贴墙的编号动作？如果不能，方针就是假的。
